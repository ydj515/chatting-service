1. 핵심 요약 — 2026-06-21 평가 대비 무엇이 바뀌었나

기존 phase7_production_readiness_assessment.md는 "Phase 7을 정의대로 완료했다고 가정"한 평가였고, 특히 B-3(핵심 metric 대량 미구현)을 Blocker로 봤다. 그 이후(2026-06-21 ~ 06-25) Phase 7 계측·검증 슬라이스가 다수 머지되어 이 부분은 상당히 해소되었다. 그러나 동시에 "계측 코드는 생겼지만 그것을 수집·저장·경보할 파이프라인이 없다"는 새로운 공백이 드러났다.

3. 새 핵심 발견 — 관측(Observability) 파이프라인이 런타임에 미배포

Phase 7에서 metric 등록 코드와 Prometheus alert rule 파일은 만들었지만, 그것을 수집·저장·시각화·경보할 구성요소가 전부 빠져 있다. 즉 release gate를 "측정 가능"하게는 만들었으나 "지속 관측·자동 판정"으로 연결하지 못했다.

Blocker급 Prometheus 수집 경로 부재

  • actuator 노출 = health,info,metrics,caches — prometheus 엔드포인트 미노출 (chat-runtime-config/.../application-docker.yml).
  • 의존성에 micrometer-core만 있고 micrometer-registry-prometheus 없음 — scrape 포맷(/actuator/prometheus) 자체가 생성되지 않음.
  • docker-compose.yml에 Prometheus / Grafana / exporter 컨테이너가 전혀 없음.

결과: infra/prometheus/rules/phase7-redis-streams-lag.rules.yml의 alert는 이를 평가할 Prometheus 서버가 없어 발화되지 않는다. 등록된 metric은 in-memory simple registry에만 존재하여 /actuator/metrics/<name>로 단건 조회만 가능하고, 시계열·대시보드·p95/p99 누적·경보가 불가능하다.

High Gateway/Fan-out 실측 metric 전무

observability_metrics.md 3.2가 release gate 입력으로 요구하는 다음 metric이 코드에 없다: active WebSocket connections, room subscription count, outbound payload bytes/sec, batch frames/sec, local delivery/sec, send queue depth, WebSocket write latency p95/p99(= fan-out 지연), slow client disconnect, reconnect rate.

chat.websocket.gateway는 metric이 아니라 @ConfigurationProperties prefix다(ChatWebSocketGatewayProperties). 특히 RoomPolicyWorker.gatewaySendQueueDepth가 default 0으로 들어가 OVERLOAD 자동 판정의 게이트웨이 신호가 실제 게이트웨이에서 측정·연결되지 않는다 → "Phase 7 release gate: hot room fan-out p95 500ms 이하" 판정 불가.

Medium PostgreSQL 관측값 placeholder 하드코딩

admin 응답의 replicaLagMs/searchP95LatencyMs가 AdminMessageRepository에서 0::bigint / NULL로 고정 반환된다. ReadReplicaLagPolicy는 fallback 판단용으로 lag를 SQL로 읽지만, partition write latency·search projection lag·archive run duration은 metric으로 등록돼 있지 않다.

4. Blocker — 이 상태로는 공개 트래픽/SLA 수용 불가

B-1 Redis 단일 인스턴스 (SPOF)

docker-compose.yml은 redis:7.2-alpine 단일 노드. Sentinel/Cluster/replica 미배포. Redis는 sequence 발급·Streams ingest·admission·presence·fanout owner lease·pub/sub의 단일 의존점이라, 노드 장애 = 메시지 수락·실시간 경로 전면 중단.

코드는 {roomId} hash tag로 Cluster를 전제했지만 실제 Cluster 배포는 없음. appendfsync everysec로 장애 시 최대 1초 ingest 유실 가능.

B-2 Object Storage 부재 + Cold Archive 미구현

AdminMessageExportWorker는 java.nio.file.Files로 로컬 디스크에 CSV 작성(exportDirectory). S3/MinIO 클라이언트·컨테이너 모두 없음.

infra/postgres/archive/archive-partitions.sh는 retention 초과 파티션을 로컬 /archive 디렉토리로 COPY 후 옵션 DROP할 뿐 S3 cold archive 경로가 없다. 멀티 인스턴스에서 export 파일이 특정 worker 로컬에만 존재 → 만료 URL 제공·감사·법적 보관 불가.

B-3 관측 가능성 (부분 해소 / 파이프라인 미배포)

계측 코드는 ingest/worker/streams/fanout-owner/ticket/admission 계열로 대폭 채워졌다(상단 3장 참조). 그러나 (a) Gateway/Fan-out 실측 metric 부재, (b) Prometheus 수집·경보 파이프라인 미배포로 인해 release gate("writer lag 3초 이하", "fan-out p95 500ms 이하" 등)를 실제로 판정·유지할 수 없다.

B-4 hot room shard 실제 분산 미적용

shard count 기본값이 모든 경로에서 1로 고정: ChatRedisProperties.shardCount = 1, RoomStorageConfigJdbcRepository.DEFAULT_SHARD_COUNT = 1, ChatServiceImpl.SHARD_COUNT = 1.

stream shard가 1이면 fanout owner도 방당 1 owner가 전 메시지를 직렬 XREADGROUP/publish → 순서는 지키지만 단일 owner가 10,000 msg/sec를 혼자 처리해야 한다. "10,000 msg/sec 60초 유지" gate를 통과하려면 shard 확장 + 방별 owner 분산 검증이 선행되어야 하나 그 구현·검증 공백이 남는다.

5. High — 공개 직후 장애/데이터/보안 리스크

항목실측 상태영향
H-1 전송 암호화 / 시크릿 infra/nginx/nginx.conf·redis.conf에 TLS/SSL/requirepass 없음. WebSocket은 ws://. .env 평문 시크릿. Vault/KMS 미연동. HMAC 키 회전 전략 없음. 공개 인터넷이면 wss+TLS 사실상 필수. 시크릿 평문 보관은 운영 보안 기준 미달.
H-2 토큰 폐기/회전(revocation) revoke|logout|refresh|blacklist 관련 코드 없음. HmacSessionTokenService는 TTL 만료만 검증. 로그아웃/제재/유출 시 즉시 무효화 경로 없음. 강제 로그아웃·키 롤오버 불가.
H-3 K8s / HPA K8s manifest/Helm 전무(k8s/helm 디렉토리 없음). Docker Compose만 존재. connection/lag/queue 기반 자동 확장 불가 → 수동 스케일. nginx stale upstream 문제도 Compose에서 잔존.
H-4 moderation / 콘텐츠 필터 금칙어·스팸·도배 필터, ban/mute/suspend 도메인 코드 없음. admission은 rate limit+slow mode만 처리. 설계 6.2/9.5의 abuse 통제 일부만 구현. 삭제는 is_deleted tombstone 수준.

7. 문서에서 "고민한 것들"의 실제 구현 여부

요청하신 핵심 항목입니다. docs/questions/chat_service_design_qa.md(24개 Q&A)와 설계서·hardening 문서에서 직접 고민한 결정들을 실제 코드와 대조했습니다.

구현 코드에서 확인됨 · 부분 일부만/우회 구현 · 미구현 고민은 했으나 코드 부재

문서에서의 고민판정실측 근거 / 비고
Q1·Q9 sequence block 선할당 폐기, roomSeq INCR 1로 수락 순서 보장구현방 id=3 정렬 역전 회귀를 피해 block allocation 제거. INCR 1 경로 확인.
Q8 fan-out 시 전체 세션 순회 금지, roomId→sessions 인덱스구현WebSocketSessionManager.sessionIdsByRoomId (ConcurrentHashMap).
Q8 Slow client bounded queue, 가득 차면 drop 후 gap fill구현BoundedOutboundSessionQueue + CloseStatus(1013, "Outbound queue full").
Q10 DB 저장 분리 — Streams append 성공 시 MESSAGE_ACCEPTED구현수락 ACK와 영속 저장 디커플링.
Q12 Writer는 consumer group, Fanout은 방별 owner lease구현FanoutOwnerLeaseService (SET NX PX + renew/release Lua + fencing token).
Q12 claimPendingIfDue() 자가 복구(XPENDING/XCLAIM)구현worker self-healing 경로 존재.
Q6·Q12 Poison pill → DLQ, deliveryCount/maxDeliveryCount구현dead_letter/deliveryCount 처리, chat.redis.stream.dead_letters metric.
Q6·Q20 멱등 저장: clientMessageId + ON CONFLICT구현PartitionedMessageRepository upsert, clientMessageId 중복 제거.
Q15 ports/adapters — MessageWritePort 추상화구현JPA→Partitioned adapter 교체 가능 구조.
Q16·Q17·Q21 Opaque cursor 페이지네이션구현admin/public cursor, numeric cursor 호환 기간 명시.
Q19 Pub/Sub 휘발성 → gap fill로 보정구현GET /chat-rooms/{id}/messages/gap?afterSeq= 경로 존재.
Q23 user-level rate limit (도배 방어)구현admission Lua의 user rate counter. MESSAGE_ADMISSION_REJECTED/429.
Q11·Q24 Replica lag fallback → Primary 우회부분ReadReplicaLagPolicy로 lag 읽어 fallback. 단, admin 응답 replicaLagMs는 0 하드코딩이라 운영 관측은 미연결.
Q2·Q4 OVERLOAD 자동 판정/강등, 다중 Pod 정책 동기화부분RoomHeatClassifier/RoomPolicyWorker 존재하나 gatewaySendQueueDepth 등 게이트웨이 실측 신호가 default 0 → OVERLOAD 입력 미연결. 정책 동기화는 per-room cache evict + TTL, Pub/Sub ROOM_STATE_CHANGED broadcast는 "향후"로 명시.
Q5 Redis MAXLEN으로 Streams OOM 방어미구현RedisMessageStreamProducer의 append에 MAXLEN/trim 미사용. HPA 지연 시 Redis OOM 방어 장치가 코드에 없음.
Q5·Q6 유실 감지: roomSeq gap audit job / Evicted 경보미구현gap scan/missing seq 감지 코드 없음. "관리자가 유실을 100% 식별"이라는 고민이 코드로는 부재.
Q22 Zombie connection 회수 heartbeat (ping-pong)미구현PingMessage/PongMessage/setHeartbeat 명시 코드 없음. 컨테이너/TCP 기본에 의존.
Q18 PgBouncer connection pooler 도입미구현load test 단계 권고였으나 미도입. worker scale-out 시 DB connection thrashing 위험.

고민 vs 구현 갭의 의미: 위 미구현 4건(MAXLEN, gap audit, heartbeat, PgBouncer)은 모두 "장애·과부하 상황에서의 생존·복구"를 다루는, 문서에서 명확히 논증했던 방어 장치다. 정상 경로 기능은 충실하나, 스스로 고민했던 극단 상황 대비책의 일부가 코드에 반영되지 않은 상태다. 특히 MAXLEN 부재 + gap audit 부재가 겹치면, OOM/장애로 ingest가 밀릴 때 (a) Redis가 무한 증식하거나 (b) 유실이 발생해도 이를 감지할 수단이 없어 위험이 복리로 커진다.

8. 권장 실행 순서 (production까지)

설계서가 이미 제시한 하이브리드 전략(토대 선행 → 계측 → 게이트)을 현재 실측 기준으로 보정한 순서입니다.

  1. 관측 파이프라인 배포 (최우선, 적은 비용 / 큰 효과) — micrometer-registry-prometheus 추가, actuator prometheus 노출, Compose에 Prometheus+Grafana 컨테이너 추가. 이미 등록된 metric과 alert rule이 즉시 살아난다.
  2. Gateway/Fan-out 실측 metric 계측 — active conn, send queue depth, write latency p95/p99, slow client, reconnect rate. 이것이 fan-out gate 판정과 OVERLOAD 자동 강등의 실제 입력이 된다.
  3. B-1 Redis HA — Compose는 Sentinel+replica(또는 단일+AOF 한시 운영 기준 명문화). hash tag는 이미 준비됨.
  4. B-2 Object Storage + cold archive — MinIO + S3 호환 추상화 1벌. archive worker DROP 전 cold archive 경로 추가, export 만료 URL 제공.
  5. 설계 고민 미구현분 보강 — Streams MAXLEN, roomSeq gap audit job, WebSocket heartbeat(ping-pong). 비용 대비 생존성 효과가 큼.
  6. B-4 shard 분산 + 10k msg/sec 검증 — shard count 동적 확장 구조 정비 후 부하 검증.
  7. 보안/운영 — H-1 TLS/wss, H-2 토큰 revocation, H-4 moderation 파이프라인.
  8. Phase 8: K8s 전환 — 모듈별 Deployment + HPA, Service/Ingress(nginx stale 문제 소멸), DR/백업(PITR, RTO/RPO).

9. 주의사항

- 이 평가는 현재 코드 실측 기준이다. 기능 테스트·로컬 데모 통과와 production 준비 완료는 다르다(단일 Redis·로컬 파일 export는 데모에선 정상 동작).

- "Blocker"는 공개 고트래픽 SLA 기준이다. 내부 PoC·제한 베타라면 B-1/B-2/B-4와 일부 High는 운영 합의로 완화 가능하다.

- Phase 7의 계측 코드가 늘었다는 사실이 "관측되고 있다"를 의미하지 않는다. 수집·경보 파이프라인이 없으면 release gate는 형식적으로만 통과한다(3장).

- 단일 VM Compose 운영 기간에는 노드 장애 = 전체 정지이므로, 이 기간 가용성 기대치를 운영 합의로 낮춰야 한다.

- shard count는 외부 계약이 아니라 운영 설정값이지만, 과거 bucket의 shard count는 유지되므로(설계 7.2) 확장 시점·전략을 미리 정해야 조회 복잡도가 통제된다.
