3. Phase 8 구현 확인 결과

8.1 완료 Observability pipeline과 Gateway metric

micrometer-registry-prometheus 의존성과 actuator prometheus endpoint가 추가되었고, Docker Compose cluster profile에 Prometheus, Alertmanager, Grafana가 들어왔다. Prometheus는 API/WebSocket/Worker/Admin을 역할별 job으로 직접 scrape한다.

  • Gateway metric: chat.websocket.gateway.connections, room.subscriptions, send.queue.depth, write.latency, outbound.bytes, slow_client.disconnects
  • Streams metric: append latency, worker records, batch latency, dead letters, group lag/pending gauge
  • Alertmanager: warning은 Slack-compatible webhook, critical/release_blocking=true는 PagerDuty로 routing. 별도 alert-smoke profile에서 synthetic firing rule 검증 가능
  • Alertmanager self-monitoring: PagerDuty notification 실패는 Alertmanager 자체 metric 기반 AlertmanagerPagerDutyNotificationFailures warning alert로 감지해 Slack으로 fallback 전송
  • Grafana datasource와 chat-release-gate.json dashboard 확인

갱신: Alertmanager/notification wiring은 Compose와 contract test로 연결됐고, synthetic firing smoke rule은 별도 profile로 격리됐다. 실제 Slack/PagerDuty secret 주입 후 alert-smoke profile로 warning Slack, critical PagerDuty delivery smoke, active alert resolve까지 확인했다. PagerDuty notification 실패 self-monitoring은 Slack warning으로 연결됐고, SLO burn-rate rule은 별도 필요하다.

8.2 완료 Redis Cluster HA

전체 Docker backend 기본 profile은 docker,redis-cluster이고, cluster profile은 Redis 6노드와 redis-cluster-init one-shot gate를 사용한다. Redis key naming은 room hash tag를 사용해 방 단위 Lua/Streams 조작이 같은 slot에 들어가도록 설계되어 있다.

  • 3 master + 3 replica 구성을 위한 6개 node 정의
  • appendonly yes, appendfsync everysec, cluster-preferred-endpoint-type hostname
  • 호스트 노출 port는 loopback에 bind되어 외부 인터페이스 노출을 줄임

잔여: Redis password/TLS는 없다. Compose 네트워크 경계에 의존하므로 실제 production에서는 ACL/TLS 또는 managed Redis 보안 설정이 필요하다.

8.3 완료 Object Storage와 cold archive

MinIO와 bucket init, S3 호환 adapter, admin export 완료 산출물 업로드, partition archive CSV/metadata 업로드가 구현되었다. DROP_AFTER_COPY=true에서 Object Storage 업로드가 비활성화되면 archive script가 중단되므로 장기 보관 없는 drop을 막는다.

  • admin export: 로컬 staging CSV 작성 후 s3://chat-archives/admin-exports/{jobId}.csv 업로드
  • download: 조회 시점에 presigned URL 발급
  • partition archive: CSV와 metadata JSON에 sha256, bytes, rowCount, object URI 기록

잔여: local staging은 여전히 worker 디스크를 사용한다. 관리형 S3/IAM/KMS/lifecycle/restore runbook은 별도 운영 작업이다.

8.4 shard 구현 완료 / gate 실측 실패 Hot room shard 분산과 release gate

메시지 수락 경로는 room_storage_configs의 current_shard_count와 fanout_shard_count를 읽고, roomSeq round-robin 기반 streamShard/fanoutShard, messageId hash 기반 writeShard를 계산한다. room-policy worker는 HOT 16, VERY_HOT/OVERLOAD 64 shard를 목표로 upsert하며, 이미 확장된 shard count는 줄이지 않는다.

  • Hot room shard 분산 자체는 구현 완료로 본다. 수락 경로의 shard 계산, room-policy shard 확장, canonical writer의 shard 보존 경로가 코드와 테스트에 있다.
  • release gate는 스크립트와 plan 단위 테스트까지 완료됐다. 기본값은 10,000 viewers, 10,000 msg/sec, 60초, stream shard 16개 이상, fanout p95 500ms 이하, stream lag 1000 이하이다.
  • 2026-06-28 01:27 KST 1차 실행은 POST http://localhost/api/ws-tickets failed: 429로 실패했다. 이때 Prometheus 기준 issue.success=60, issue.rate_limited_ip=1, consume.success=60이었다.
  • 이후 docker-compose.yml 공통 앱 환경에 CHAT_AUTH_WEB_SOCKET_TICKET_RATE_LIMIT_*를 추가하고, gate 실행용으로 PER_IP=20000, PER_USER=20을 주입해 API 컨테이너 env 반영을 확인했다.
  • 2026-06-28 01:40 KST 재실행은 429를 넘겼지만 fetch failed로 실패했다. Prometheus 기준 issue.success=5104, consume.success=5104까지 진행했고 DB 기준 room 2 member는 5,105명까지 생성됐다.
  • 실패는 shard/fanout 부하 본문 이전의 viewer 준비 단계에서 발생했다. nginx/app 컨테이너는 실패 후에도 healthy였고 명시적인 app exception은 없었지만, nginx 런타임 기본값은 worker_connections 1024로 확인되어 10k WebSocket gate 전용 proxy capacity 조정이 다음 확인 지점이다.
  • worker와 WebSocket Gateway가 분리된 배포에서는 gateway send queue depth가 cross-process로 room-policy worker에 전달되지 않는다. 현재 같은 프로세스에 session manager가 없으면 0으로 폴백한다.

8.5 완료 Moderation과 사용자 제재

DB 기반 moderation rule과 user sanction이 추가되었다. 메시지 전송 전에 user sanction과 content moderation을 확인하고, 거부 시 MessageModerationRejectedException으로 API/WebSocket에 명시적으로 실패를 반환한다.

  • GLOBAL/ROOM CONTAINS rule, REJECT action
  • ROOM MUTE/BAN, GLOBAL SUSPEND 제약
  • admin API: rule 생성/수정/비활성화, sanction 생성/조회/해제
  • 거부 metric: chat.message.moderation.rejected

잔여: 현재 필터는 단순 contains 중심이다. 스팸 패턴, 링크/첨부/이미지 안전성, appeal workflow, moderator UI는 production 운영 범위에서 추가 검토가 필요하다.

8.6 완료 Token revocation과 global suspend

session token은 v1 payload에 issuedAt을 포함하고, Redis token denylist와 user revoke-after marker를 확인한다. POST /users/logout는 bearer token을 revoke하며, GLOBAL SUSPEND 생성 후에는 transaction commit 이후 user token revoke와 Redis pub/sub force logout을 실행한다.

  • token 단위 revoke: chat:auth:session:revoked:token:{sha256}
  • user 단위 revoke-after: chat:auth:session:revoked:user:{userId}
  • 인증 시 revocation store 오류는 fail-closed 처리
  • force logout 이벤트 수신 gateway는 해당 user의 local WebSocket session을 4003 Session revoked로 종료

잔여: Redis pub/sub force logout은 best-effort다. 놓친 이벤트가 있어도 이후 인증은 막히지만 이미 열린 연결 즉시 종료는 보장하지 못한다.

8.7 구현, 활성화, 테스트 완료 MAXLEN, roomSeq gap audit, heartbeat

Redis Streams append는 기본 MAXLEN 1000000 approximate trim을 사용하고, canonical chat_messages의 room_seq gap을 aggregate metric으로 노출하는 audit worker와 warning rule이 추가되었다. WebSocket Gateway는 ping/pong heartbeat로 zombie connection을 닫는다.

항목 상태 남은 확인
Redis Streams MAXLEN 구현 및 단위 테스트 완료 운영 값 1000000이 실제 10k 부하에서 충분한지 running-stack으로 조정 필요
roomSeq gap audit 구현, metric, Prometheus rule, 기본 role, runtime scrape 완료 기본 Compose cluster에서 room-seq-gap-audit role이 활성화됨. worker actuator와 Prometheus에서 chat_room_seq_gap_* scrape 확인. 대규모 운영에서는 중복 스캔 방지를 위해 전용 worker 분리 가능
WebSocket heartbeat 구현, Compose env, runtime ping smoke 완료 기본 Compose cluster에 heartbeat enabled/interval/poll/timeout env가 명시됨. verify-websocket-heartbeat.mjs로 실제 서버 ping frame 수신 확인. 장시간 soak와 no-pong timeout rehearsal은 release rehearsal 항목으로 유지
  • RedisMessageStreamProducer: XAddOptions.maxlen(...).approximateTrimming(...)
  • RoomSeqGapAuditRepository: 최근 lookback window에서 lag(room_seq) 기반 gap 집계
  • RoomSeqGapAuditMetrics: rooms, missing_sequences, max_width, scanned_rooms gauge
  • WebSocketHeartbeatScheduler: 기본 10초 poll, 30초 ping, 90초 timeout

잔여: 장시간 soak, proxy timeout, no-pong timeout cleanup, 실제 장애 리허설은 release rehearsal 항목으로 남는다. 기능 미구현, 기본 role 미반영, running-stack scrape/ping 미확인은 해소됐다.

4. 남은 운영 리스크

Alertmanager 온콜 delivery smoke는 2026-06-28 04:44 KST 실행 기준으로 해소 처리한다. 실제 Slack/PagerDuty secret을 주입한 뒤 AlertmanagerSmokeWarning은 Slack, AlertmanagerSmokeCritical은 PagerDuty로 라우팅됐고, 사용자 확인 기준 실제 수신까지 완료됐다. 결과 증거: Slack warning 수신, PagerDuty critical incident 수신, 담당자명 제거. Alertmanager 로그에는 firing/resolved notify 오류가 없었으며 smoke alert active count도 0으로 정리됐다.
우선순위 리스크 상태와 영향 권장 조치
Blocker TLS/wss와 secret 관리 Nginx는 HTTP 80만 listen한다. WebSocket 안내도 ws://이고, Redis/MinIO/app secret은 개발용 기본값 중심이다. TLS termination, wss, 운영 CORS/origin 제한, secret manager 또는 ExternalSecret, HMAC key rotation 절차 추가

Redis failover, archive upload, moderation smoke도 running stack에서는 아직 미확인이다. nginx worker connection/file limit, load generator HTTP 실패 원인, app/proxy connection budget을 조정한 뒤 hot-room gate, Redis node kill/failover, MinIO upload, moderation smoke, heartbeat soak를 재실행해 artifact 보관
High K8s/HPA/DR 부재 역할별 실행 모듈은 준비됐지만 K8s/Helm manifest는 없다. HPA, rolling update, pod disruption budget, PITR/RTO/RPO가 정의되지 않았다. Phase 9에서 Deployment/Service/Ingress/HPA, Redis/S3/PostgreSQL managed resource, backup/restore rehearsal 작성
High room-policy의 cross-process signal gap worker가 별도 프로세스이면 Gateway send queue depth가 0으로 폴백한다. OVERLOAD 판정의 gateway 압력 신호가 운영에서 빠질 수 있다. Gateway queue depth를 Redis/metric snapshot으로 publish하고 room-policy worker가 aggregate하도록 보강
Medium PgBouncer/DB connection 운영 토대 없음 다중 app/worker scale-out 시 PostgreSQL connection 수와 failover handling이 병목이 될 수 있다. PgBouncer 또는 managed pooler, max connection budget, app별 Hikari cap 정의

6. 복잡도와 평가 한계

항목 복잡도 설명
저장소 근거 수집 시간 O(F + L), 공간 O(R) F는 확인한 파일 수, L은 검색/열람한 라인 수, R은 보고서에 보관한 근거 수다.
테스트 실행 시간 O(T), 공간 O(A) T는 실행된 테스트/태스크 수, A는 Gradle/Node 테스트 산출물 크기다.
Docker gate 실행 시간 O(C + V + M), 공간 O(C + V) C는 기동 컨테이너 수, V는 WebSocket viewer 수, M은 전송 메시지 수다. 이번 실행은 V=10000, M=600000 목표였으나 2차 재시도 기준 5,104 ticket 발급/소비 후 viewer 준비 단계에서 실패했다.
Alertmanager delivery smoke 시간 O(R + N), 공간 O(N) R은 Alertmanager route 개수, N은 synthetic alert 개수다. 이번 실행에서는 warning/critical 2개 alert로 Slack/PagerDuty delivery와 resolved 정리를 확인했다.
HTML 보고서 생성 시간 O(S), 공간 O(S) S는 생성된 HTML 문서 크기다.

8. 권장 실행 순서

  1. nginx worker connection/file limit과 load generator HTTP 실패 원인을 보강한 뒤 phase8-hot-room-release-gate.mjs를 재실행해 결과 JSON과 Prometheus snapshot을 보관한다.
  2. heartbeat no-pong timeout/cleanup, Redis node kill/failover, worker kill, MinIO upload failure를 포함한 장애 리허설을 실행한다.
  3. TLS/wss, 운영 origin/CORS, secret manager, HMAC key rotation을 정리한다.
  4. alert-smoke profile을 staging CI job으로 자동화하고 silence 리허설을 정례화한다.
  5. Gateway send queue depth를 cross-process signal로 publish해 room-policy OVERLOAD 판정을 완성한다.
  6. K8s/Helm 또는 동등한 배포 매니페스트, HPA, PDB, backup/restore runbook을 작성한다.
  7. 제한 베타로 낮은 트래픽을 받아 SLO와 비용을 관찰한 뒤 공개 production으로 확장한다.

9. 주의사항

- 이 평가는 코드와 로컬 구성의 readiness 평가다. 실제 클라우드 managed resource, 네트워크, 디스크, TLS, IAM, 장애 전파 시간은 검증하지 않았다.

- MAXLEN은 Redis OOM 방어용 backpressure이며 메시지 보존 보장이 아니다. gap audit와 alert, archive/backup runbook이 함께 있어야 의미가 있다.

- Redis pub/sub force logout은 best-effort다. 사용자 revoke marker는 이후 인증을 막지만, 이미 열린 연결 즉시 종료는 event 유실 시 놓칠 수 있다.

- Compose cluster는 production-like 검증 환경에 가깝지만, production 배포 모델 자체는 아니다. K8s 또는 managed platform의 rollout, HPA, secret, storage 정책이 별도로 필요하다.

- Slack/PagerDuty delivery smoke는 현재 제공된 secret 기준으로 확인했다. escalation policy 변경, key rotation, staging CI 자동화는 별도 운영 관리가 필요하다.

- 10k release gate는 실제 실행했지만 1차는 ticket 발급 429, 2차는 5,104 ticket 발급/소비 후 fetch failed로 실패했다. 공개 production 판정에는 viewer 준비, 부하 본문, Prometheus threshold까지 통과한 artifact가 필요하다.
